# Service Worker 自主运行机制修复指南

## 问题描述
之前的实现中，Service Worker 需要页面持续发送心跳和任务指令才能维持运行。当页面失活后，心跳停止，Service Worker 的循环就会中断。

## 解决方案
修改了 Service Worker 架构，实现真正的自主运行机制：

### 1. 核心改动

#### Service Worker 端 (`public/service-worker.js`)
- **新增自主监控机制**：`startAutonomousMonitoring()` 和 `stopAutonomousMonitoring()`
- **任务配置保存**：将任务配置保存到 `window.lastTaskConfig`，用于自主重启
- **内部心跳机制**：`startInternalMonitoring()` 提供独立的状态监控
- **自动重启逻辑**：当检测到循环任务停止时，自动重启发送流程

#### 页面端 (`public/sender.js`)
- **自主运行提示**：循环模式启动时显示自主运行通知
- **延迟时间调整**：将循环延迟从3秒调整为15秒，减少服务器压力
- **停止逻辑优化**：停止时清理自主运行提示和状态

### 2. 工作原理

1. **启动阶段**：
   - 用户点击"开始传送"并选择循环模式
   - 页面发送任务配置给 Service Worker
   - Service Worker 保存配置并启动自主监控
   - 显示自主运行提示（5秒后自动消失）

2. **运行阶段**：
   - Service Worker 每30秒检查一次任务状态
   - 如果检测到循环任务停止，自动重启
   - 页面失活或关闭不影响 Service Worker 继续运行
   - 最大自主重启次数：999次

3. **停止阶段**：
   - 用户点击停止或任务完成
   - Service Worker 停止自主监控
   - 清理所有相关状态和提示

### 3. 关键特性

- **真正的后台运行**：页面失活后 Service Worker 继续执行
- **自动故障恢复**：检测到中断时自动重启
- **用户友好提示**：清晰的状态提示和通知
- **资源优化**：合理的检查间隔和重启限制

### 4. 使用方法

1. 选择要发送的文件或素材
2. 勾选"循环模式"
3. 点击"开始传送"
4. 看到"自主运行模式"提示后，可以安全地切换应用或关闭浏览器
5. Service Worker 将在后台持续运行循环发送任务

### 5. 技术细节

- **监控间隔**：30秒检查一次任务状态
- **重启延迟**：检测到停止后等待5秒再重启
- **循环延迟**：每轮发送间隔15秒
- **最大重启次数**：999次（防止无限重启）

## 测试验证

1. 启动循环模式发送任务
2. 观察自主运行提示出现
3. 切换到其他应用或最小化浏览器
4. 等待一段时间后回到页面
5. 确认任务仍在继续执行

## 注意事项

- 自主运行仅在循环模式下启用
- 非循环模式仍然依赖页面保持活跃
- 浏览器完全关闭可能影响 Service Worker 运行
- 建议保持至少一个浏览器标签页打开以确保最佳性能
