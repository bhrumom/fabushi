# 搜索方案对比

## 架构对比

### 旧方案：文件系统搜索

```
用户请求
   ↓
Worker接收
   ↓
读取manifest.json
   ↓
遍历所有txt文件 ←─────┐
   ↓                   │
读取文件内容 (ASSETS)   │ 重复N次
   ↓                   │
字符串匹配             │
   ↓                   │
收集结果 ──────────────┘
   ↓
排序
   ↓
返回结果

时间: 2-5秒
内存: 高
并发: 差
```

```
用户请求
   ↓
Worker接收
   ↓
D1 SQL查询 (FTS5索引)
   ↓
返回结果

时间: 50-200ms
内存: 低
并发: 优秀
```

## 代码对比

### 旧方案代码（简化）

```javascript
// 需要读取多个文件
async function handleSearch(request, env) {
  const files = await getAllTextFiles(env);
  const results = [];
  
  // 遍历所有文件
  for (const file of files) {
    // 每次都要读取文件
    const content = await readTextFile(env, file.path);
    
    // 字符串匹配
    if (content.includes(query)) {
      results.push({...});
    }
  }
  
  return results;
}
```

**问题**:
- ❌ 每次搜索都要读取文件
- ❌ O(n*m) 时间复杂度
- ❌ 无法分页
- ❌ 无法筛选

### 新方案代码（简化）

```javascript
// 一次SQL查询搞定
async function handleSearch(request, env, db) {
  const { results } = await db.prepare(`
    SELECT * FROM text_contents
    WHERE title LIKE ? OR content LIKE ?
    ORDER BY title_match DESC
    LIMIT ? OFFSET ?
  `).bind(pattern, pattern, limit, offset).all();
  
  return results;
}
```

**优势**:
- ✅ 一次查询完成
- ✅ O(log n) 时间复杂度（索引）
- ✅ 支持分页
- ✅ 支持筛选
